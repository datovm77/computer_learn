# 06-代码修改：小步快跑、限制范围与验收标准

本章目标：把 Claude Code 用在实际改代码时，学会先计划、限制范围、保持最小 diff，并用明确验收标准收尾。

## 使用场景

适合修一个小 bug、补一个小功能、调整一个组件行为、修改接口字段、更新文档或增加一条校验逻辑。核心原则是：一次只做一件事，先让 Claude Code 说明计划，再允许它编辑指定文件。

高风险提醒：改代码时尤其要警惕删除文件、批量重构、改权限、安装依赖、提交代码、联网下载包、处理密钥。除非你明确要求，否则不要让 Claude Code 自动提交代码或执行会改数据库、改生产配置的命令。

普通终端命令：

```bash
cd /path/to/project
claude
```

Claude Code 对话输入：

```markdown
我要修复一个小问题：保存用户资料时，空昵称应该被拒绝。
请先只读分析相关文件，给出修改计划。不要先改代码。
限制范围：优先只改 src/user/profile.ts 和对应测试文件。
验收标准：新增或更新测试；相关测试通过；diff 保持最小。
```

## 推荐提示词模板

Claude Code 对话输入：

```markdown
请按“小步修改”流程工作。

目标：
[写清楚要改的行为]

限制范围：
- 优先只修改：[文件或目录]
- 不要修改：[明确禁止的文件或目录]
- 不要做整库格式化、批量重构、依赖升级或无关清理

流程：
1. 先只读分析相关代码
2. 给出计划和预计会改的文件
3. 等我确认后再编辑
4. 修改后展示关键 diff 摘要
5. 运行最小必要验证

验收标准：
- [行为标准]
- [测试标准]
- [兼容性标准]
- [文档或错误信息标准，如需要]

高风险动作必须先问我：删除文件、安装依赖、改权限、提交代码、联网、读取密钥。
```

确认计划后再输入：

Claude Code 对话输入：

```markdown
计划可以。请只修改你刚才列出的文件，保持最小 diff。修改完成后运行你建议的最小测试命令。
```

## 错误示范

Claude Code 对话输入：

```markdown
这个功能有点问题，你看着改，顺便优化一下代码质量。
```

问题：目标不清、边界不清，“顺便优化”会引入无关改动。

Claude Code 对话输入：

```markdown
把用户模块都重写成更好的写法。
```

问题：没有验收标准，也没有说明不改变哪些行为，风险远超普通修复。

普通终端命令：

```bash
claude -p "修复所有 lint 和 test 问题，并提交代码"
```

问题：自动修复所有问题范围过大，提交代码属于高风险动作，应先审查 diff。

## 正确示范

Claude Code 对话输入：

```markdown
请修复订单金额为 0 时仍可提交的问题。
限制范围：只考虑 src/order/submit.ts 和 tests/order/submit.test.ts。
请先解释当前逻辑为什么放过了 0，再给出最小修改方案，等我确认。
```

Claude Code 对话输入：

```markdown
请实现方案 A。不要改公共接口，不要格式化整文件。完成后运行订单提交相关测试即可。
```

普通终端命令：

```bash
npm test -- tests/order/submit.test.ts
```

如果测试命令可能不存在，先让 Claude Code 识别项目脚本。

Claude Code 对话输入：

```markdown
请先查看 package.json 和测试目录，判断应该运行哪条最小测试命令。不要安装依赖。
```

## 练习任务

1. 选择一个很小的代码修改，例如补充字段校验或修正文案。
2. 写出限制范围和验收标准。
3. 让 Claude Code 先读代码并给计划，不要立刻改。
4. 确认计划后让它修改。
5. 让它运行最小测试或说明为什么无法运行。
6. 审查 diff，要求它解释每个改动对应哪个验收标准。

进阶练习：故意加入一个“不要修改某文件”的限制，观察 Claude Code 是否遵守。

## 验收标准

- 修改前有计划，计划中列出预计文件和验证方式。
- 实际修改文件没有超出你给定的范围。
- diff 只包含实现目标所需内容，没有顺手重构或整库格式化。
- 至少运行了最小必要验证，或明确说明无法运行的原因。
- 高风险动作均经过确认，没有自动删除、安装、提交、联网或处理密钥。
- 最终总结能对应到最初验收标准。
