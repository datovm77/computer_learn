# 12-命令行自动化：print 模式、管道、脚本与批处理

Claude Code 不只可以交互式对话，也可以通过命令行自动化执行一次性任务。`claude -p` 常被称为 print 模式：把提示词交给 Claude Code，等待它输出结果，然后进程结束。它适合摘要、分类、生成报告、检查 diff、批量处理文本和接入脚本。

⚠️ 自动化会降低人工停顿，尤其要谨慎处理删除文件、批量重构、改权限、安装依赖、提交代码、联网和密钥。脚本中应默认只读，只有经过明确审批才允许写入或执行危险命令。

## 使用场景

适合使用 print 模式的场景：

- 把 `git diff` 输入给 Claude Code，让它生成代码审查摘要。
- 对日志、测试输出、构建错误做结构化分析。
- 在 CI 或本地脚本中生成变更说明、发布说明、检查清单。
- 批量处理 Markdown、JSON 或纯文本文件。
- 用管道把其他命令输出交给 Claude Code 总结。

不适合使用 print 模式的场景：

- 需要多轮澄清的复杂需求。
- 需要人工确认的删除、回退、权限修改、依赖安装、提交代码。
- 可能访问生产数据、密钥或隐私数据的任务。
- 无人值守的大规模重构。

自动化边界建议：

- 默认只读：让命令输出报告，而不是直接修改文件。
- 显式输入：把要分析的文件或 diff 通过管道传入。
- 显式输出：要求 Markdown、JSON 或固定格式，便于后续脚本处理。
- 显式风险：在提示词中写明禁止删除文件、联网、提交代码、处理密钥。
- 小批量运行：先对 1 个文件验证，再扩大到目录。

## 推荐提示词模板

普通终端命令：

```bash
git diff --staged | claude -p "请用中文审查这段已暂存 diff。只输出：高风险问题、测试缺口、是否建议提交。不要修改文件。"
```

普通终端命令：

```bash
cat app.log | claude -p "请分析日志中的错误模式。输出 Markdown 表格：时间、错误、可能原因、建议动作。不要联网，不要修改文件。"
```

普通终端命令：

```bash
claude -p "请生成一个安全检查清单，用于代码提交前检查：密钥、危险命令、依赖安装、权限变化、测试结果。输出中文 Markdown。"
```

普通终端命令：

```bash
for f in docs/*.md; do
  claude -p "请检查下面 Markdown 是否有错别字和结构问题。只输出建议，不要改文件：$(printf '%s' "$f")"
done
```

上面的示例只把文件名传给模型，不会自动读取文件内容。更稳妥的做法是显式传入内容：

普通终端命令：

```bash
for f in docs/*.md; do
  printf '\nFILE: %s\n' "$f"
  claude -p "请检查这个 Markdown 文件内容，只输出建议，不要修改文件。" < "$f"
done
```

Claude Code 对话输入：

```markdown
请帮我设计一个使用 claude -p 的脚本，用于审查 git diff。
要求：
- 默认只读。
- 不提交代码。
- 不安装依赖。
- 不联网。
- 如果 diff 中包含密钥疑似内容，必须高亮提醒。
- 输出脚本和使用说明。
```

## 错误示范

普通终端命令：

```bash
claude -p "扫描整个项目，删除无用文件，自动提交"
```

哪里不对：

- 删除文件和提交代码都是高风险动作。
- 没有定义“无用文件”的判断标准。
- 没有人工确认环节。
- 没有测试和 diff 审核。

普通终端命令：

```bash
find . -type f | claude -p "把所有文件都重构到最佳状态"
```

哪里不对：

- 范围过大。
- 管道只传了文件列表，不代表 Claude Code 已获得每个文件的完整上下文。
- “最佳状态”不可验收。
- 批量重构风险高。

普通终端命令：

```bash
cat .env | claude -p "帮我看看配置有没有问题"
```

哪里不对：

- `.env` 可能包含密钥、token、数据库密码。
- 处理密钥属于高风险内容。
- 应先脱敏或只传配置键名。

普通终端命令：

```bash
claude -p "联网搜索最新方案并直接改项目"
```

哪里不对：

- 联网和写入同时发生，缺少审核。
- 没有来源要求。
- 没有文件边界。

## 正确示范

普通终端命令：

```bash
git diff -- src/ tests/ | claude -p "你是只读代码审查助手。请审查输入的 diff。

限制：
- 不要修改文件。
- 不要提交代码。
- 不要安装依赖。
- 不要联网。
- 如果发现密钥、token、证书、权限变化、删除文件、批量重构，请列为高风险。

输出：
1. 阻塞问题
2. 非阻塞建议
3. 测试建议
4. 是否可以进入人工复核"
```

普通终端命令：

```bash
npm test 2>&1 | claude -p "请分析测试失败原因。只根据输入内容判断，不要联网，不要修改文件。输出：失败测试、根因猜测、下一步排查命令。"
```

普通终端命令：

```bash
git status --short | claude -p "请根据 git status 输出判断本轮改动范围是否清晰。不要建议提交，只输出需要人工确认的文件。"
```

普通终端命令：

```bash
printf '%s\n' "src/auth/login.ts" "tests/auth/login.test.ts" |
claude -p "请根据文件清单生成一个中文验收清单。不要假设文件内容，不要修改文件。"
```

PowerShell 普通终端命令：

```powershell
git diff -- src tests | claude -p "请只读审查这个 diff，输出中文 Markdown。高风险项包括密钥、删除文件、权限变化、依赖安装、提交或回退。"
```

脚本模板：

普通终端命令：

```bash
#!/usr/bin/env bash
set -euo pipefail

git diff -- "$@" | claude -p "请只读审查输入的 diff。
禁止：修改文件、删除文件、安装依赖、联网、提交代码、处理密钥。
如果输入为空，请说明没有 diff。
输出中文 Markdown。"
```

## 练习任务

练习 1：审查暂存区

普通终端命令：

```bash
git diff --staged | claude -p "请审查暂存区 diff。输出阻塞问题、测试建议、提交前检查。不要修改文件。"
```

练习 2：分析测试输出

普通终端命令：

```bash
npm test 2>&1 | claude -p "请分析测试输出，列出最可能的第一个失败原因和建议排查命令。不要联网。"
```

练习 3：生成发布说明草稿

普通终端命令：

```bash
git log --oneline -10 | claude -p "请根据提交标题生成中文发布说明草稿。不要编造未出现的功能。"
```

练习 4：设计安全批处理

Claude Code 对话输入：

```markdown
请设计一个批量检查 Markdown 文件标题层级的方案。
要求：
- 先只读检查。
- 不自动改文件。
- 输出发现列表。
- 扩展到自动修复前必须有人工确认。
```

## 验收标准

学完本章，你应该能：

- 解释 `claude -p` 适合一次性、可脚本化、边界清晰的任务。
- 使用管道把 `git diff`、日志、测试输出传给 Claude Code。
- 在提示词中明确禁止删除、批量重构、改权限、安装依赖、提交代码、联网和处理密钥。
- 区分“传入文件名”和“传入文件内容”。
- 为批处理脚本设计只读默认行为和人工确认点。
- 知道什么时候应该改用交互式 Claude Code，而不是 print 模式。
