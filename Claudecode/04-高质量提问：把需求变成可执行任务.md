# 高质量提问：把需求变成可执行任务

Claude Code 擅长执行清晰任务。模糊需求会导致它猜范围、猜目标、猜验收标准，最后产生难以审查的变更。高质量提问的目标，是把“我想要什么”改写成“在什么背景下、修改哪里、不能做什么、怎样算完成”。

高风险提醒：当需求涉及删除文件、批量重构、修改权限、安装依赖、提交代码、联网访问、处理密钥时，必须在提示词中明确写出是否允许，并要求 Claude Code 先等待确认。

## 使用场景

适合以下任务：

- 把一句模糊需求改成 Claude Code 可执行任务。
- 限定修改范围，避免越界改动。
- 要求 Claude Code 先给计划，再改文件。
- 明确输出格式，方便你审查。
- 为 bug 修复、功能开发、测试补充、文档更新写任务说明。

高质量任务通常包含：

- 背景：为什么要做，当前问题是什么。
- 目标：希望达成什么结果。
- 范围：允许查看或修改哪些文件、目录、模块。
- 限制：禁止做什么，例如不安装依赖、不联网、不删除文件。
- 验收：如何判断完成，例如测试通过、文档更新、页面行为改变。
- 输出：你希望 Claude Code 返回计划、diff 摘要、测试结果还是提交信息。

## 推荐提示词模板

Claude Code 对话输入：

```markdown
请先制定计划，等我确认后再修改文件。

背景：
- 当前问题是：这里写具体问题。
- 我已经观察到：这里写报错、现象或用户反馈。

目标：
- 希望实现：这里写清楚结果。

范围：
- 允许查看：这里写目录或文件。
- 允许修改：这里写目录或文件。

限制：
- 不删除文件。
- 不批量重构。
- 不修改权限。
- 不安装依赖。
- 不联网。
- 不提交代码。
- 不处理真实密钥。

验收标准：
- 这里写测试、命令、页面行为或文档标准。

输出格式：
- 先给计划。
- 修改后列出文件清单、diff 摘要、验证结果和剩余风险。
```

Claude Code 对话输入：

```markdown
请把下面的模糊需求改写成适合 Claude Code 执行的任务提示词。不要修改文件。

模糊需求：
“登录有点问题，帮我修一下。”

请输出：
- 你需要追问的问题
- 一个保守版本提示词
- 一个信息完整后的正式提示词
```

## 错误示范

Claude Code 对话输入：

```markdown
页面不好看，改漂亮点。
```

问题：

- “不好看”和“漂亮”不可验收。
- 没有指定页面、风格、范围。
- 没有说明是否允许改组件库、安装依赖或批量重构。

Claude Code 对话输入：

```markdown
帮我修 bug。
```

问题：

- 没有 bug 现象、复现步骤、期望结果。
- Claude Code 只能猜测。
- 容易改到无关文件。

Claude Code 对话输入：

```markdown
这个仓库你全权处理，最后提交。
```

问题：

- 范围过大。
- 提交代码是高风险动作。
- 没有限制删除文件、联网、依赖安装、权限变更和密钥处理。

## 正确示范

Claude Code 对话输入：

```markdown
请先不要修改文件，先帮我定位问题。

背景：
- 用户在登录页输入正确账号密码后，页面仍停留在登录页。
- 浏览器控制台显示 401。

目标：
- 找出 401 的可能来源。
- 给出最小修改方案。

范围：
- 优先查看 `src/auth`、`src/api`、登录页组件和相关测试。
- 不查看无关业务模块，除非你说明理由。

限制：
- 不删除文件。
- 不批量重构。
- 不安装依赖。
- 不联网。
- 不提交代码。
- 不处理真实密钥；如果需要环境变量，只使用 `.env.example` 中的占位名。

输出格式：
- 先列调查计划。
- 再列你需要查看的文件。
- 最后给出候选原因和下一步建议。
```

Claude Code 对话输入：

```markdown
请修改 README 的本地运行说明。

背景：
- 新同事按照 README 无法找到测试命令。

目标：
- 在 README 中补充启动、测试、常见失败原因三部分。

范围：
- 只允许修改 `README.md`。

限制：
- 不改代码。
- 不安装依赖。
- 不联网。
- 不提交代码。

验收标准：
- README 中能看到启动命令和测试命令。
- 不出现真实密钥或个人路径。

输出格式：
- 修改后列出改动摘要和建议我运行的普通终端命令。
```

## 从模糊到清晰

模糊需求：

Claude Code 对话输入：

```markdown
帮我优化这个项目。
```

清晰任务：

Claude Code 对话输入：

```markdown
请只读分析当前项目，找出 3 个最值得优化的小任务，不要修改文件。

每个任务请包含：
- 问题描述
- 影响范围
- 建议修改文件
- 风险等级
- 验收标准

限制：
- 不删除文件。
- 不批量重构。
- 不安装依赖。
- 不联网。
- 不提交代码。
```

## 练习任务

1. 写下 3 个你平时会说的模糊需求。
2. 为每个需求补充背景、目标、范围、限制、验收标准和输出格式。
3. 把其中一个提示词交给 Claude Code，让它先只读分析。
4. 根据 Claude Code 的追问继续补全信息。
5. 修改前要求 Claude Code 复述任务边界。

Claude Code 对话输入：

```markdown
请评审我的任务提示词是否足够清晰。不要修改文件。

请检查：
- 背景是否足够
- 目标是否可执行
- 范围是否明确
- 限制是否覆盖高风险动作
- 验收标准是否可检查
- 输出格式是否便于审查
```

## 验收标准

完成本章后，你应该能做到：

- 能把模糊需求拆成背景、目标、范围、限制、验收和输出格式。
- 能明确禁止或确认删除文件、批量重构、改权限、安装依赖、提交代码、联网和处理密钥。
- 能要求 Claude Code 先计划、再执行、最后汇报。
- 能写出适合 bug 修复、文档更新和小功能开发的任务提示词。
- 能判断一个提示词是否会导致越界修改。
