# 15-综合实战：用 Claude Code 完成一个真实小项目

本章把前面的能力串起来：用 Claude Code 完成一个真实小项目，从需求澄清、只读分析、计划、实现、测试、总结到复盘。重点不是追求一次性自动完成，而是建立可审查、可回退、可协作的工程流程。

示例项目：给一个已有 Web 项目增加“任务标签筛选”功能。你可以替换成自己的项目，例如命令行工具、文档站、脚本集合或小型后端 API。

高风险提醒：综合实战通常会涉及多文件修改、测试命令、依赖判断、提交代码甚至联网查文档。删除文件、批量重构、改权限、安装依赖、提交代码、联网、处理密钥都必须单独确认。

## 使用场景

适合按本章流程处理的任务：

- 新增一个小功能。
- 修复一个跨文件 bug。
- 给已有功能补测试。
- 重构一个边界明确的小模块。
- 为项目补一段文档或自动化脚本。

不适合直接进入实现的任务：

- “重写整个项目”。
- “把技术栈换掉”。
- “自动修好所有问题”。
- “删除无用文件并提交”。
- “连接生产系统验证”。

完整流程：

1. 需求澄清：定义目标、非目标、验收标准。
2. 只读分析：读取项目结构、相关文件、测试入口。
3. 计划确认：列出修改文件、风险、测试命令。
4. 小步实现：只改计划内文件。
5. 本地验证：运行最小必要测试。
6. 审核 diff：检查越界修改和高风险内容。
7. 总结交付：列出改动、测试结果、遗留风险。
8. 复盘改进：记录提示词、命令和流程问题。

协作提示：本章默认用单个 Claude Code 会话完成小项目。只有当任务能拆成清晰独立模块时，才考虑子 Agent 做只读分析、专项实现或审核。需要多个 Claude Code 实例互相沟通和自协调时，才考虑 agent teams；它是实验功能，默认关闭，且更适合有清楚文件所有权或独立 worktree 的并行工作。

## 推荐提示词模板

阶段 1：需求澄清

Claude Code 对话输入：

```markdown
我们要给现有项目增加“任务标签筛选”功能。

请先只做需求澄清，不要读取大量文件，不要修改文件。
请输出：
- 你需要确认的问题。
- 可能的验收标准。
- 可能涉及的风险。

高风险动作提醒：删除文件、批量重构、改权限、安装依赖、提交代码、联网、处理密钥都必须先确认。
```

阶段 2：只读分析

Claude Code 对话输入：

```markdown
现在进入只读分析。

目标：找到任务列表、筛选状态、测试入口相关文件。

限制：
- 不修改文件。
- 不安装依赖。
- 不联网。
- 不提交代码。
- 不读取密钥文件。

输出：
1. 相关文件
2. 当前数据流
3. 建议修改点
4. 建议测试命令
5. 风险和不确定点
```

阶段 3：计划确认

Claude Code 对话输入：

```markdown
请基于只读分析给出实现计划。

计划必须包含：
- 只允许修改的文件清单。
- 每个文件要改什么。
- 不改什么。
- 测试命令。
- 回滚策略。
- 高风险动作是否涉及：删除文件、批量重构、改权限、安装依赖、提交代码、联网、处理密钥。

在我确认前不要实现。
```

阶段 4：实现

Claude Code 对话输入：

```markdown
我确认计划。现在开始实现。

写入边界：
- 只修改计划中列出的文件。
- 不格式化无关文件。
- 不删除文件。
- 不安装依赖。
- 不提交代码。
- 不联网。
- 不处理密钥。

实现完成后，请列出修改路径和建议验证命令。
```

阶段 5：验证

普通终端命令：

```bash
npm test -- --runInBand
```

如果项目不是 Node.js，请替换成实际测试命令，例如：

普通终端命令：

```bash
pytest
```

普通终端命令：

```bash
cargo test
```

阶段 6：审查

普通终端命令：

```bash
git status --short
git diff --stat
git diff --check
```

Claude Code 对话输入：

```markdown
请只读审查当前 diff。

重点：
- 是否只修改了计划内文件。
- 是否出现密钥、删除、权限变化、依赖安装、联网调用、提交或回退。
- 是否有测试缺口。
- 是否有用户可见行为变化未记录。

不要修改文件。
```

阶段 7：交付总结

Claude Code 对话输入：

```markdown
请生成本次任务总结。
包含：
- 修改文件。
- 行为变化。
- 测试结果。
- 未完成事项。
- 风险提醒。
- 建议提交信息。
不要执行 git commit。
```

## 错误示范

Claude Code 对话输入：

```markdown
帮我做一个任务标签筛选功能，你自己看着办，做完直接提交。
```

问题：

- 没有验收标准。
- 没有只读分析阶段。
- 没有文件边界。
- “直接提交”是高风险动作。

Claude Code 对话输入：

```markdown
为了实现标签筛选，可以顺便把状态管理重构掉，把旧组件删了。
```

问题：

- 功能开发和批量重构混在一起。
- 删除文件属于高风险动作。
- 影响范围扩大，测试压力增加。

普通终端命令：

```bash
npm install some-random-filter-library
```

问题：

- 未先评估是否需要依赖。
- 依赖安装会联网并修改 lockfile。
- 小功能通常应优先使用现有能力。

Claude Code 对话输入：

```markdown
测试失败你就随便改到通过。
```

问题：

- “随便改”没有边界。
- 可能掩盖真实 bug。
- 可能修改测试来迎合错误实现。

## 正确示范

一次完整对话流程示例：

Claude Code 对话输入：

```markdown
我要给任务列表增加标签筛选。

验收标准：
- 用户可以选择一个标签，只显示含该标签的任务。
- 可以选择“全部”恢复完整列表。
- 刷新页面后不要求保留筛选状态。
- 不新增依赖。
- 不修改认证、后端 API 或构建配置。

请先只读分析相关文件和测试入口，不要修改文件。
```

Claude Code 对话输入：

```markdown
请根据分析结果制定计划。
计划中必须列出允许修改的文件。
如果需要修改计划外文件，必须先停止并说明。
```

Claude Code 对话输入：

```markdown
确认计划。请实现。
限制：
- 只修改 src/components/TaskList.tsx、src/components/TagFilter.tsx、tests/TaskList.test.tsx。
- 不安装依赖。
- 不删除文件。
- 不提交代码。
- 不联网。
```

普通终端命令：

```bash
npm test -- TaskList
```

普通终端命令：

```bash
git diff -- src/components/TaskList.tsx src/components/TagFilter.tsx tests/TaskList.test.tsx
```

Claude Code 对话输入：

```markdown
请只读审核这些改动是否满足验收标准。
不要修改文件。
如果发现问题，请按阻塞和非阻塞分类。
```

正确交付总结示例：

markdown：

```markdown
本次完成：
- 新增标签筛选控件。
- 任务列表根据所选标签过滤。
- 补充 TaskList 标签筛选测试。

验证：
- npm test -- TaskList 通过。
- git diff --check 通过。

未做：
- 未持久化筛选状态。
- 未修改后端 API。
- 未新增依赖。

风险：
- 仅覆盖任务列表组件测试，未做端到端测试。
```

## 练习任务

练习 1：为自己的项目选题

Claude Code 对话输入：

```markdown
请帮我把当前项目中适合 Claude Code 练习的小任务列出来。
要求：
- 只读分析。
- 每个任务控制在 1 到 3 个文件。
- 不涉及生产密钥、部署、删除文件、依赖安装。
- 输出任务名称、可能文件、验收标准。
```

练习 2：写需求和非目标

Claude Code 对话输入：

```markdown
我想做一个小功能：[填入你的功能]。
请帮我整理：
- 目标。
- 非目标。
- 验收标准。
- 需要确认的问题。
不要修改文件。
```

练习 3：制定计划

Claude Code 对话输入：

```markdown
请只读分析并制定实现计划。
计划必须包含允许修改的文件清单、测试命令、高风险动作检查。
不要实现。
```

练习 4：实现后复盘

Claude Code 对话输入：

```markdown
请根据本轮对话和 diff 做复盘。
输出：
- 哪些提示词有效。
- 哪些地方边界不清。
- 下次应提前确认什么。
- 是否出现删除、批量重构、改权限、安装依赖、提交代码、联网、处理密钥风险。
不要修改文件。
```

## 验收标准

完成本章后，你应该能够：

- 用 Claude Code 把真实小项目拆成需求、分析、计划、实现、测试、审查和总结。
- 在实现前定义目标、非目标和验收标准。
- 要求 Claude Code 先只读分析，再修改文件。
- 明确允许修改的文件清单，并禁止越界修改。
- 在测试和审查阶段识别删除、批量重构、改权限、安装依赖、提交代码、联网、处理密钥风险。
- 生成可交付的任务总结和提交信息草稿，但不让 Claude Code 擅自提交。
- 复盘提示词和流程，形成可复用工作流。
