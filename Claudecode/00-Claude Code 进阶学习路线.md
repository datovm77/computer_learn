# Claude Code 进阶学习路线

本章说明这套教材的学习顺序、使用方法和安全边界。默认你已经安装 Claude Code，目标不是重新安装，而是把 Claude Code 当成一个可以读项目、改文件、运行命令、配合 Git 工作的编程助理来掌握。

建议按顺序学习：

1. `00`：建立学习路线和安全习惯。
2. `01`：理解 Claude Code 的工作方式、工具、权限和环境自检。
3. `02`：补齐终端、路径、项目根目录和项目结构基础。
4. `03`：掌握 Git、diff、commit 与回退风险。
5. `04`：把模糊需求改写成 Claude Code 能执行的任务。

高风险提醒：让 Claude Code 删除文件、批量重构、修改权限、安装依赖、提交代码、联网访问、处理密钥或凭证前，必须明确范围、确认影响，并先看计划和 diff。

## 使用场景

适合以下情况：

- 你已经能打开终端并运行 `claude`，但不知道如何系统练习。
- 你想让 Claude Code 处理真实项目，而不是只问概念题。
- 你经常遇到“它改了很多文件但我看不懂”的情况。
- 你想建立从提问、执行、检查、提交到复盘的完整流程。

不适合以下用法：

- 直接把生产仓库交给 Claude Code 大范围修改。
- 在不了解项目根目录和 Git 状态时要求批量重构。
- 把 API Key、SSH 私钥、数据库密码直接贴进对话。

普通终端命令：

```bash
pwd
git status
claude
```

Claude Code 对话输入：

```markdown
请先阅读当前项目结构，不要修改文件。请告诉我这个项目的入口文件、依赖文件、测试目录和你建议我先学习的 3 个文件。
```

## 学习路径

第一阶段：会定位。你要能回答“我在哪个目录”“这个项目根目录是什么”“Claude Code 可能看到哪些文件”。

第二阶段：会约束。你要能把任务写清楚，包括背景、目标、范围、限制、验收标准和输出格式。

第三阶段：会检查。你要能看懂 `git diff`，知道 Claude Code 改了什么、为什么改、是否越界。

第四阶段：会交付。你要能让 Claude Code 运行测试、整理变更摘要，并在你确认后再提交。

第五阶段：会复盘。你要能把成功提示词、踩坑原因和项目约定沉淀成笔记或 `CLAUDE.md`。

## 推荐提示词模板

Claude Code 对话输入：

```markdown
你是我的 Claude Code 实战教练。请基于当前目录帮我制定一个 30 分钟练习计划。

背景：
- 我已经安装 Claude Code。
- 我想学习进阶实战，而不是安装教程。

目标：
- 理解当前项目结构。
- 练习只读分析、局部修改、查看 diff、运行检查。

范围：
- 只分析当前项目。
- 不删除文件，不批量重构，不安装依赖，不提交代码。

输出格式：
- 先列学习步骤。
- 再列每一步要输入的命令或提示词。
- 最后列验收标准。
```

Claude Code 对话输入：

```markdown
请为我执行一个安全的学习任务：只读取当前项目，不修改文件。请找出项目根目录、主要语言、依赖文件、测试命令候选项，并说明你依据了哪些文件。
```

## 错误示范

Claude Code 对话输入：

```markdown
帮我把这个项目优化一下，顺便提交。
```

问题：

- 没有说明优化目标。
- 没有限定修改范围。
- “顺便提交”属于高风险动作，提交代码前必须先看 diff 和测试结果。
- 没有说明是否允许安装依赖、联网、删除文件或批量重构。

Claude Code 对话输入：

```markdown
这个项目有点乱，你直接重构所有文件。
```

问题：

- 批量重构风险很高。
- 没有验收标准。
- 容易引入大量难以审查的变更。

## 正确示范

Claude Code 对话输入：

```markdown
请先只读分析当前项目，不要修改文件。

请完成：
1. 判断项目根目录是否正确。
2. 找出入口文件、依赖文件、测试目录。
3. 找出最适合新手阅读的 5 个文件。
4. 列出你建议的第一个小修改任务。

限制：
- 不删除文件。
- 不安装依赖。
- 不联网。
- 不提交代码。

输出：
- 用表格列出文件路径和判断依据。
- 最后给出下一步建议。
```

Claude Code 对话输入：

```markdown
请修改一个最小范围的问题：只更新 README 中的运行说明，不改代码文件。

执行前请先给计划；执行后请给：
- 修改过的文件列表
- 关键 diff 摘要
- 是否需要我运行命令验证
```

## 练习任务

1. 在一个非生产项目中启动 Claude Code。
2. 让 Claude Code 只读分析项目结构。
3. 根据它的分析，选择一个只改文档或只改测试的小任务。
4. 要求 Claude Code 先给计划，再修改。
5. 修改后使用普通终端命令查看 diff。

普通终端命令：

```bash
git status
git diff
```

Claude Code 对话输入：

```markdown
请根据刚才的修改生成一份变更摘要，但不要提交代码。摘要需要包含：改了什么、为什么改、如何验证、还有哪些风险。
```

## 验收标准

完成本章后，你应该能做到：

- 能说清这套教材每一章解决什么问题。
- 能解释为什么要先分析项目结构，再让 Claude Code 修改。
- 能在高风险动作前明确要求 Claude Code 等待确认。
- 能识别删除文件、批量重构、改权限、安装依赖、提交代码、联网、处理密钥这些高风险场景。
- 能写出一个包含背景、目标、范围、限制、验收标准的 Claude Code 任务。
