# 09-重构专题：整理代码但不改变行为

这一章讲怎么让 Claude Code 帮你整理代码结构、命名和重复逻辑，同时明确”不改变行为”，并用测试守住边界。

## 使用场景

适合拆分过长函数、提取重复逻辑、改善命名、移动内部辅助函数、简化条件分支、整理模块边界。重构不是“顺手改功能”，它的验收重点是外部行为不变、测试仍然通过、diff 可审查。

⚠️ 重构很容易扩大范围。不要让 Claude Code 批量重构全仓库、格式化整库、删除文件、改权限、安装依赖、升级依赖、提交代码、联网查资料或处理密钥。涉及公共 API、数据库迁移、权限逻辑、认证逻辑时，要分阶段做。

普通终端命令：

```bash
claude --permission-mode plan
```

Claude Code 对话输入：

```markdown
我想重构订单价格计算逻辑，但不改变任何外部行为。
请先只读分析相关代码和测试，输出分阶段计划。
不要修改代码。
```

## 推荐提示词模板

Claude Code 对话输入：

```markdown
请按“行为不变重构”流程工作。

重构目标：
[例如：拆分过长函数、提取重复校验、改善命名]

限制范围：
- 只考虑：[文件或目录]
- 不要修改：[公共接口、数据库 schema、配置、无关模块]
- 不要格式化整库或做批量机械替换

流程：
1. 先找出现有测试和可运行验证命令
2. 说明当前行为边界
3. 给出分阶段重构计划，每阶段都可独立验证
4. 等我确认后只执行第一阶段
5. 每阶段结束都总结 diff 和验证结果

验收标准：
- 外部行为不变
- 现有测试通过
- diff 小且可审查
- 没有引入新依赖
- 高风险动作先问我
```

阶段完成后继续：

Claude Code 对话输入：

```markdown
第一阶段通过。请继续第二阶段，仍然保持行为不变。每次只改你计划中的文件。
```

## 错误示范

Claude Code 对话输入：

```markdown
这个项目代码太乱了，帮我全面重构。
```

哪里不对：范围不可控，容易制造无法审查的大 diff。

Claude Code 对话输入：

```markdown
把所有文件格式化并顺便升级到最新写法。
```

哪里不对：整库格式化和升级写法会混在一起，无法判断行为是否改变。

普通终端命令：

```bash
claude -p "重构整个 src 目录，自动提交"
```

哪里不对：批量重构和自动提交都是高风险动作，应该分阶段审查。`-p` 是非交互模式，执行完直接退出，没有审查 diff 的机会。

## 正确示范

Claude Code 对话输入：

```markdown
请只读分析 src/pricing/calculate.ts。
目标：把 180 行的 calculatePrice 拆成几个私有辅助函数。
要求：不改变导出函数签名，不改变错误信息，不改变舍入规则。
请先找相关测试和风险点，再给计划。
```

Claude Code 对话输入：

```markdown
请执行第一阶段：只提取优惠券校验辅助函数。不要改其它逻辑。完成后运行 pricing 相关测试。
```

普通终端命令：

```bash
npm test -- tests/pricing.test.ts
```

Claude Code 对话输入：

```markdown
请对比重构前后的行为边界：输入、输出、异常、日志、副作用。只基于代码和测试，不要猜。
```

## 练习任务

1. 找一个 80 行以上但测试覆盖较好的函数。
2. 让 Claude Code 找到相关测试和行为边界。
3. 要求它给出 2 到 3 个阶段的重构计划。
4. 只批准第一阶段。
5. 运行相关测试并审查 diff。
6. 如果 diff 过大，要求 Claude Code 回到更小的步骤。

进阶练习：让 Claude Code 为重构前缺失的关键行为先补一个回归测试，再开始重构。

## 验收标准

- 重构前找到了相关测试或说明了测试缺口。
- 计划按阶段执行，每阶段都能独立验证。
- 公共接口、错误信息、数据格式和副作用保持不变，除非你明确批准。
- 没有整库格式化、批量替换或无关清理。
- 每阶段都有 diff 摘要和验证结果。
- 高风险动作没有未经确认执行。
