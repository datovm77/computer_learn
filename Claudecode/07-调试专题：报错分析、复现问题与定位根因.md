# 07-调试专题：报错分析、复现问题与定位根因

这一章讲怎么让 Claude Code 按”复现→收集证据→定位根因→最小修复→验证”的顺序调试，避免看到报错就盲改。

## 使用场景

适合处理测试失败、运行时报错、构建失败、接口异常、页面行为不符合预期、偶发问题或线上日志分析。调试时最重要的是可复现：把错误命令、输入数据、错误栈、期望行为、实际行为交给 Claude Code。

⚠️ 调试时不要让 Claude Code 直接删除数据、清空缓存目录、改权限、重装依赖、升级依赖、提交代码、联网搜索错误、复制密钥或连接生产环境。需要运行命令时，先确认是否会写数据库、调用外部服务或修改文件。

普通终端命令：

```bash
npm test -- tests/login.test.ts
```

Claude Code 对话输入：

```markdown
这个命令失败了。请先分析错误，不要修改代码。

复现命令：`npm test -- tests/login.test.ts`

错误现象：
[粘贴关键错误栈，不要粘贴密钥]

期望行为：
[写明应该发生什么]

请输出：复现步骤、最可能根因、需要查看的文件、最小修复方案和验证命令。
```

## 推荐提示词模板

Claude Code 对话输入：

```markdown
请按调试流程处理这个问题。

问题描述：
[现象]

复现方式：
普通终端命令：[命令]

错误信息：
[错误栈或日志摘要]

约束：
- 先不要改代码
- 不要删除文件、改权限、安装依赖、联网、提交代码或读取密钥
- 如果需要运行命令，先说明命令目的、副作用和预计耗时

请先输出：
1. 错误栈逐层解释
2. 已知事实和假设
3. 需要检查的文件
4. 根因定位计划
5. 最小修复方案
6. 验证命令
```

确认定位后再输入：

Claude Code 对话输入：

```markdown
请按最小修复方案修改。只改与根因直接相关的文件。不要顺手重构。修改后运行复现命令验证。
```

## 错误示范

Claude Code 对话输入：

```markdown
报错了，帮我修一下。
```

哪里不对：没有命令、错误栈、期望行为，Claude Code 只能猜。

Claude Code 对话输入：

```markdown
测试挂了，你把所有测试都改到通过。
```

哪里不对：可能把测试改成适应错误行为，掩盖真正根因。

普通终端命令：

```bash
rm -rf node_modules && npm install && npm test
```

哪里不对：删除和安装依赖都是高风险动作，可能掩盖问题或引入新变量。应先确认是否必要。

## 正确示范

Claude Code 对话输入：

```markdown
请分析这个失败，不要修改代码。

普通终端命令：pytest tests/test_checkout.py::test_coupon_expired

错误摘要：
AssertionError: expected 400, got 200

期望行为：过期优惠券不能被使用，接口应返回 400。
请追踪优惠券校验逻辑，找出为什么过期时间没有生效。
```

Claude Code 对话输入：

```markdown
根因确认后，请只做最小修复，并补一个回归测试证明过期优惠券会被拒绝。不要修改其它优惠券逻辑。
```

普通终端命令：

```bash
pytest tests/test_checkout.py::test_coupon_expired
```

## 练习任务

1. 找一个能稳定复现的失败测试或构造一个小 bug。
2. 把复现命令、错误摘要、期望行为写给 Claude Code。
3. 要求 Claude Code 先解释错误栈，不要修改代码。
4. 让它列出根因定位计划和需要查看的文件。
5. 确认后让它做最小修复。
6. 运行同一条复现命令，并让 Claude Code 说明为什么现在通过。

进阶练习：让 Claude Code 区分“直接原因”和“根因”，并说明是否需要补回归测试。

## 验收标准

- 有明确复现步骤和复现命令。
- Claude Code 在修改前解释了错误栈、事实、假设和定位计划。
- 修复直接对应根因，没有大范围重写或无关格式化。
- 原失败用例通过，并尽量增加回归测试。
- 验证命令和结果被记录在最终总结中。
- 没有执行未经确认的删除、安装、改权限、提交、联网、生产访问或密钥处理。
