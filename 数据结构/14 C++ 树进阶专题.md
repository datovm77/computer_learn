## Part 1 ｜ 线索二叉树（Threaded Binary Tree）

## 第一章 线索二叉树的概念

### 1.1 为什么需要线索二叉树？

回顾一个重要结论：**含有 $n$ 个节点的二叉链表中，有 $n + 1$ 个空指针。**

```
          A
         / \
        B   C
       / \
      D   E

节点数 n = 5
指针总数 = 2n = 10
使用的指针 = n - 1 = 4（即边数）
空指针 = 10 - 4 = 6 个
```

这些空指针没有存储有效信息。能否利用它们来**加速遍历**？

另一个问题是：在普通二叉链表中，要找某个节点的**中序前驱**或**中序后继**，通常需要重新做一次中序遍历（或用栈模拟），时间复杂度 O(n)。如果能直接通过指针找到前驱/后继，遍历和定位都会更高效。

**线索二叉树就是解决这两个问题的方案。**

### 1.2 线索化的基本思想

**规则**：

- 如果节点的**左指针为空**，就让它指向该节点的**中序前驱**
- 如果节点的**右指针为空**，就让它指向该节点的**中序后继**

这些指向前驱/后继的指针称为**线索（Thread）**。

但这引入一个问题：怎么区分一个指针是**指向孩子**还是**指向前驱/后继（线索）**？

**解决方案**：为每个节点增加两个标志位 `ltag` 和 `rtag`：

| 标志 | 值 = 0 | 值 = 1 |
|------|--------|--------|
| `ltag` | left 指向**左孩子** | left 指向**中序前驱**（线索）|
| `rtag` | right 指向**右孩子** | right 指向**中序后继**（线索）|

### 1.3 节点结构

```cpp
struct ThreadNode {
    int data;
    ThreadNode* left;
    ThreadNode* right;
    int ltag;  // 0 = 左孩子，1 = 前驱线索
    int rtag;  // 0 = 右孩子，1 = 后继线索

    ThreadNode(int val)
        : data(val), left(nullptr), right(nullptr), ltag(0), rtag(0) {}
};
```

### 1.4 线索化的图示

以中序线索化为例：

```
原始二叉树：              中序遍历顺序：D B E A F C
         A
        / \
       B   C
      / \ /
     D  E F

中序线索二叉树（虚线 = 线索，实线 = 真实孩子）：

         A
        / \
       B   C
      / \ /
     D  E F

节点 D：无左孩子 → ltag=1, left 指向 null（D 是中序第一个，无前驱）
        无右孩子 → rtag=1, right 指向 B（D 的中序后继是 B）

节点 E：无左孩子 → ltag=1, left 指向 B（E 的中序前驱是 B）
        无右孩子 → rtag=1, right 指向 A（E 的中序后继是 A）

节点 F：无左孩子 → ltag=1, left 指向 A（F 的中序前驱是 A）
        无右孩子 → rtag=1, right 指向 C（F 的中序后继是 C）

节点 B：有左孩子 D → ltag=0
        有右孩子 E → rtag=0

节点 C：有左孩子 F → ltag=0
        无右孩子 → rtag=1, right 指向 null（C 是中序最后一个，无后继）

节点 A：有左孩子 B → ltag=0
        有右孩子 C → rtag=0
```

用表格更清楚地展示：

| 节点 | left | ltag | data | rtag | right |
|------|------|------|------|------|-------|
| D | null | 1 | D | 1 | →B |
| B | →D | 0 | B | 0 | →E |
| E | →B | 1 | E | 1 | →A |
| A | →B | 0 | A | 0 | →C |
| F | →A | 1 | F | 1 | →C |
| C | →F | 0 | C | 1 | null |

### 1.5 中序线索化的实现

线索化本质上就是一次中序遍历，在遍历过程中修改空指针。

```cpp
class ThreadedBinaryTree {
private:
    ThreadNode* root;
    ThreadNode* prev;  // 指向中序遍历中的前一个节点

    void inorderThread(ThreadNode* node) {
        if (node == nullptr) return;

        // 递归线索化左子树
        inorderThread(node->left);

        // 处理当前节点
        if (node->left == nullptr) {
            node->left = prev;      // 左指针指向前驱
            node->ltag = 1;
        }
        if (prev != nullptr && prev->right == nullptr) {
            prev->right = node;     // 前驱的右指针指向当前节点（后继）
            prev->rtag = 1;
        }

        prev = node;  // 更新 prev

        // 递归线索化右子树
        inorderThread(node->right);
    }

public:
    ThreadedBinaryTree(ThreadNode* r) : root(r), prev(nullptr) {}

    void createInorderThread() {
        prev = nullptr;
        if (root != nullptr) {
            inorderThread(root);
            // 处理最后一个节点的右指针
            if (prev->right == nullptr) {
                prev->rtag = 1;
            }
        }
    }
};
```

### 1.6 利用线索进行中序遍历

线索化后，不需要栈也不需要递归，就能进行中序遍历：

```cpp
// 找中序遍历的第一个节点：一路向左走到底
ThreadNode* firstNode(ThreadNode* node) {
    if (node == nullptr) return nullptr;
    while (node->ltag == 0) {
        node = node->left;
    }
    return node;
}

// 找节点的中序后继
ThreadNode* nextNode(ThreadNode* node) {
    if (node->rtag == 1) {
        return node->right;   // 线索直接指向后继
    }
    // 右子树的最左节点就是后继
    return firstNode(node->right);
}

// 中序遍历（无需栈和递归！）
void inorderTraversal(ThreadNode* root) {
    for (ThreadNode* node = firstNode(root);
         node != nullptr;
         node = nextNode(node)) {
        cout << node->data << " ";
    }
    cout << endl;
}
```

**这就是线索二叉树的价值**——O(1) 空间复杂度的中序遍历！

### 1.7 线索二叉树总结

| 维度 | 普通二叉链表 | 线索二叉链表 |
|------|-------------|-------------|
| 空指针利用 | 浪费 n+1 个空指针 | 利用空指针存线索 |
| 找中序前驱/后继 | O(n)，需遍历 | O(1)，直接读指针或走到右子树最左 |
| 中序遍历空间 | O(h)（递归/栈） | O(1) |
| 缺点 | — | 插入/删除需维护线索，实现更复杂 |

> 先序线索化和后序线索化原理类似，只是在先序/后序遍历过程中设置线索。实际应用中**中序线索化**最常见。

## Part 2 ｜ 并查集（Union-Find / Disjoint Set）

## 第二章 并查集的概念

### 2.1 什么是并查集？

**并查集**是一种用于管理**不相交集合**（Disjoint Sets）的数据结构，支持两种核心操作：

| 操作 | 含义 |
|------|------|
| **Find(x)** | 查找元素 x 所属的集合（用集合的“代表元素”来标识） |
| **Union(x, y)** | 将 x 所在集合和 y 所在集合**合并**为一个集合 |

生活类比：

```
场景：判断两个人是否属于同一个朋友圈

初始：每个人各自一个圈子
  {A}  {B}  {C}  {D}  {E}  {F}

A 和 B 是朋友 → Union(A, B)
  {A, B}  {C}  {D}  {E}  {F}

B 和 C 是朋友 → Union(B, C)
  {A, B, C}  {D}  {E}  {F}

D 和 E 是朋友 → Union(D, E)
  {A, B, C}  {D, E}  {F}

A 和 D 是朋友吗？ → Find(A) ≠ Find(D) → 不是
A 和 C 是朋友吗？ → Find(A) == Find(C) → 是
```

### 2.2 用树（森林）实现并查集

并查集用**森林**表示——每棵树代表一个集合，**树的根**就是这个集合的代表元素。

存储方式：**双亲表示法**（数组存储），每个元素记录它的父节点。

```cpp
int parent[MAX_SIZE];

// 初始化：每个元素自成一个集合（自己是自己的根）
void init(int n) {
    for (int i = 0; i < n; i++) {
        parent[i] = i;  // 也可以初始化为 -1
    }
}
```

```
初始状态（每人自成一棵树）：
parent: [0, 1, 2, 3, 4, 5]
         A  B  C  D  E  F

表示 6 棵只有根的树：
  A   B   C   D   E   F
```

### 2.3 基本操作

#### Find（查找）

沿着 parent 指针一路向上走，直到找到根节点（parent[x] == x 的节点）。

```cpp
int find(int x) {
    while (parent[x] != x) {
        x = parent[x];
    }
    return x;
}
```

#### Union（合并）

先找到两个元素的根，如果不同，就让其中一个根指向另一个根。

```cpp
void unionSets(int x, int y) {
    int rootX = find(x);
    int rootY = find(y);
    if (rootX != rootY) {
        parent[rootX] = rootY;  // 把 x 的根挂到 y 的根下面
    }
}
```

**合并过程图示**：

```
Union(A, B)：Find(A)=A, Find(B)=B，让 A 的根指向 B
  B          C   D   E   F
  |
  A

Union(B, C)：Find(B)=B, Find(C)=C，让 B 的根指向 C
    C        D   E   F
    |
    B
    |
    A

Union(D, E)：Find(D)=D, Find(E)=E
    C        E   F
    |        |
    B        D
    |
    A

此时 parent 数组：
parent: [1, 2, 2, 4, 4, 5]
         A  B  C  D  E  F

Find(A)：A→B→C，返回 C
Find(D)：D→E，返回 E
Find(A) ≠ Find(D)，所以 A 和 D 不在同一集合
```

### 2.4 问题：树可能退化

如果每次合并都是把一棵小树挂到另一棵上，可能形成一条很长的链：

```
最坏情况：依次 Union(0,1), Union(1,2), Union(2,3), ...

  0 → 1 → 2 → 3 → 4 → 5 → ...

Find 操作可能需要 O(n) 时间。
```

因此需要优化。

### 2.5 优化一：按秩合并（Union by Rank）

**思想**：合并时，总是把**矮的树**挂到**高的树**下面，尽量保持树的平衡。

用 `rank` 数组记录每棵树的高度（秩）：

```cpp
int parent[MAX_SIZE];
int rankArr[MAX_SIZE];

void init(int n) {
    for (int i = 0; i < n; i++) {
        parent[i] = i;
        rankArr[i] = 0;
    }
}

void unionByRank(int x, int y) {
    int rootX = find(x);
    int rootY = find(y);
    if (rootX == rootY) return;

    if (rankArr[rootX] < rankArr[rootY]) {
        parent[rootX] = rootY;          // 矮树挂到高树下
    } else if (rankArr[rootX] > rankArr[rootY]) {
        parent[rootY] = rootX;
    } else {
        parent[rootY] = rootX;          // 等高时任选一个
        rankArr[rootX]++;               // 新树高度 +1
    }
}
```

按秩合并保证树高不超过 $O(\log n)$。

### 2.6 优化二：路径压缩（Path Compression）

**思想**：在 Find 操作时，顺便把路径上所有节点**直接挂到根**下面，后续查找会明显变快。

```cpp
int find(int x) {
    if (parent[x] != x) {
        parent[x] = find(parent[x]);  // 递归，把所有节点直接连到根
    }
    return parent[x];
}
```

**路径压缩过程图示**：

```
Find(A) 之前：               Find(A) 之后：
      E                          E
      |                       / | \ \
      D                      A  B  C  D
      |
      C
      |
      B
      |
      A

A→B→C→D→E 变成了 A→E, B→E, C→E, D→E
下次 Find(A) 只需一步就能到达根。
```

**非递归版本**（避免深度过大时栈溢出）：

```cpp
int find(int x) {
    int root = x;
    while (parent[root] != root) {
        root = parent[root];
    }
    // 路径压缩：把沿途所有节点直接挂到根下
    while (parent[x] != root) {
        int next = parent[x];
        parent[x] = root;
        x = next;
    }
    return root;
}
```

### 2.7 两种优化同时使用

当按秩合并和路径压缩**同时使用**时，$m$ 次操作（Find + Union）的总时间复杂度接近 $O(m \cdot \alpha(n))$，其中 $\alpha$ 是**阿克曼函数的反函数**，增长极其缓慢。对于所有实际规模的输入，$\alpha(n) \leq 4$，可以视为**常数**。

### 2.8 完整实现

```cpp
#include <iostream>
#include <vector>
using namespace std;

class UnionFind {
private:
    vector<int> parent;
    vector<int> rank;

public:
    UnionFind(int n) : parent(n), rank(n, 0) {
        for (int i = 0; i < n; i++) {
            parent[i] = i;
        }
    }

    int find(int x) {
        if (parent[x] != x) {
            parent[x] = find(parent[x]);
        }
        return parent[x];
    }

    void unite(int x, int y) {
        int rootX = find(x);
        int rootY = find(y);
        if (rootX == rootY) return;

        if (rank[rootX] < rank[rootY]) {
            parent[rootX] = rootY;
        } else if (rank[rootX] > rank[rootY]) {
            parent[rootY] = rootX;
        } else {
            parent[rootY] = rootX;
            rank[rootX]++;
        }
    }

    bool connected(int x, int y) {
        return find(x) == find(y);
    }
};

int main() {
    // 0=A, 1=B, 2=C, 3=D, 4=E, 5=F
    UnionFind uf(6);

    uf.unite(0, 1);  // A-B 是朋友
    uf.unite(1, 2);  // B-C 是朋友
    uf.unite(3, 4);  // D-E 是朋友

    cout << "A 和 C 是朋友？" << (uf.connected(0, 2) ? "是" : "否") << endl;
    cout << "A 和 D 是朋友？" << (uf.connected(0, 3) ? "是" : "否") << endl;

    uf.unite(2, 3);  // C-D 是朋友（间接连通了 A 和 D）

    cout << "合并后，A 和 D 是朋友？" << (uf.connected(0, 3) ? "是" : "否") << endl;

    return 0;
}
```

**输出**：

```
A 和 C 是朋友？是
A 和 D 是朋友？否
合并后，A 和 D 是朋友？是
```

|----------|------|
| 连通性判断 | 判断图中两个节点是否连通 |

## Part 3 ｜ 平衡二叉树（AVL Tree）

## 第三章 AVL 树的概念

### 3.1 为什么需要 AVL 树？

BST 的典型问题是：如果插入序列接近有序，树会退化成链表，查找效率从 O(log n) 退化到 O(n)。

```
插入序列 {10, 20, 30, 40, 50}：

BST 退化成右斜链：      理想的平衡 BST：
  10                        30
    \                      /  \
     20                  20    40
      \                 /       \
       30              10        50
        \
         40
          \
           50

查找 50：5 次比较        查找 50：3 次比较
```

**AVL 树**（以发明者 Adelson-Velsky 和 Landis 命名）是一种**自平衡的 BST**，它保证树在插入和删除后不会过度倾斜。

### 3.2 定义

**AVL 树**是一棵 BST，且满足：**每个节点的左子树高度与右子树高度之差的绝对值不超过 1**。

这个差值称为**平衡因子（Balance Factor, BF）**：

$$BF(node) = height(left) - height(right)$$

AVL 树要求所有节点的 $BF \in \{-1, 0, 1\}$。

```
合法的 AVL 树：          不合法（节点 A 的 BF = 2）：
      B  (BF=0)              A  (BF=2)
     / \                    /
    A   C  (BF=0)          B  (BF=1)
 (BF=0)                  /
                         C  (BF=0)
```

### 3.3 AVL 节点结构

```cpp
struct AVLNode {
    int data;
    AVLNode* left;
    AVLNode* right;
    int height;  // 以该节点为根的子树高度

    AVLNode(int val) : data(val), left(nullptr), right(nullptr), height(1) {}
};

int getHeight(AVLNode* node) {
    return node ? node->height : 0;
}

int getBalance(AVLNode* node) {
    return node ? getHeight(node->left) - getHeight(node->right) : 0;
}

void updateHeight(AVLNode* node) {
    node->height = 1 + max(getHeight(node->left), getHeight(node->right));
}
```

### 3.4 失衡的四种情况与旋转

插入一个节点后，从插入点沿路径回溯到根，第一个 $|BF| > 1$ 的节点就是**失衡节点**。根据失衡的模式，分为四种情况：

| 失衡类型 | 描述 | 旋转方式 |
|----------|------|----------|
| **LL**（左左） | 在失衡节点的**左孩子的左子树**插入 | **右旋** |
| **RR**（右右） | 在失衡节点的**右孩子的右子树**插入 | **左旋** |
| **LR**（左右） | 在失衡节点的**左孩子的右子树**插入 | 先**左旋**左孩子，再**右旋**失衡节点 |
| **RL**（右左） | 在失衡节点的**右孩子的左子树**插入 | 先**右旋**右孩子，再**左旋**失衡节点 |

> **判断口诀**：看失衡节点（BF 绝对值 > 1 的节点）和导致失衡的插入方向。两次方向相同（LL 或 RR）→ 单旋转；两次方向不同（LR 或 RL）→ 双旋转。

### 3.5 右旋（Right Rotation）—— 解决 LL 失衡

**场景**：在 A 的左孩子 B 的左子树上插入，导致 A 失衡。

```
失衡前（LL 型）：

        A  (BF=2)
       /
      B  (BF=1)
     /
    C     ← 新插入

右旋操作：让 B 成为新的根

旋转前：          旋转后：
    A                B
   / \             /   \
  B   T4          C     A
 / \             / \   / \
C   T3          T1 T2 T3 T4
/ \
T1 T2
```

**右旋代码**：

```cpp
AVLNode* rightRotate(AVLNode* y) {
    AVLNode* x = y->left;
    AVLNode* T = x->right;

    // 旋转
    x->right = y;
    y->left = T;

    // 更新高度（先更新 y，再更新 x，因为 y 现在是 x 的孩子）
    updateHeight(y);
    updateHeight(x);

    return x;  // x 成为新的根
}
```

**详细图解**：

```
        y                x
       / \             /   \
      x   T4    →    T1     y
     / \                   / \
    T1  T2               T2  T4
         ↑
    这个子树 T2 从 x 的右孩子变成了 y 的左孩子
```

### 3.6 左旋（Left Rotation）—— 解决 RR 失衡

**场景**：在 A 的右孩子 B 的右子树上插入，与 LL 完全镜像。

```
失衡前（RR 型）：

    A  (BF=-2)
     \
      B  (BF=-1)
       \
        C     ← 新插入

左旋操作：让 B 成为新的根

旋转前：          旋转后：
    y                x
   / \             /   \
  T1  x    →     y     T4
     / \        / \
    T2  T4     T1  T2
```

**左旋代码**：

```cpp
AVLNode* leftRotate(AVLNode* y) {
    AVLNode* x = y->right;
    AVLNode* T = x->left;

    // 旋转
    x->left = y;
    y->right = T;

    // 更新高度
    updateHeight(y);
    updateHeight(x);

    return x;
}
```

### 3.7 LR 双旋 —— 解决 LR 失衡

**场景**：在 A 的左孩子 B 的**右子树**上插入。单纯右旋解决不了，需要先对 B 做一次左旋，转化为 LL 型，再对 A 做右旋。

```
失衡前（LR 型）：

      A  (BF=2)
     /
    B  (BF=-1)
     \
      C     ← 新插入

第 1 步：对 B 左旋（把 LR 转化为 LL）

      A               A
     /               /
    B       →       C
     \             /
      C           B

第 2 步：对 A 右旋

      A                C
     /        →      /   \
    C                B     A
   /
  B
```

**代码**（在 insert 中处理）：

```cpp
// LR 型：先左旋左孩子，再右旋当前节点
node->left = leftRotate(node->left);
return rightRotate(node);
```

### 3.8 RL 双旋 —— 解决 RL 失衡

与 LR 完全镜像：先对右孩子右旋，再对当前节点左旋。

```
失衡前（RL 型）：

    A  (BF=-2)
     \
      B  (BF=1)
     /
    C     ← 新插入

第 1 步：对 B 右旋（转化为 RR）

    A               A
     \               \
      B       →       C
     /                 \
    C                   B

第 2 步：对 A 左旋

    A                C
     \      →      /   \
      C           A     B
       \
        B
```

**代码**：

```cpp
// RL 型：先右旋右孩子，再左旋当前节点
node->right = rightRotate(node->right);
return leftRotate(node);
```

### 3.9 四种旋转的完整总结

```
LL 型（左左）→ 右旋
    z                y
   / \             /   \
  y   D    →     x     z
 / \            / \   / \
x   C          A   B C   D
/ \
A  B

RR 型（右右）→ 左旋
  z                  y
 / \               /   \
A   y      →     z     x
   / \          / \   / \
  B   x        A   B C   D
     / \
    C   D

LR 型（左右）→ 先左旋 y，再右旋 z
    z              z              x
   / \           / \            /   \
  y   D   →    x   D    →    y     z
 / \          / \            / \   / \
A   x        y   C          A  B  C  D
   / \      / \
  B   C    A   B

RL 型（右左）→ 先右旋 y，再左旋 z
  z            z                x
 / \          / \             /   \
A   y   →   A   x    →     z     y
   / \         / \         / \   / \
  x   D      B   y       A   B C   D
 / \             / \
B   C           C   D
```

### 3.10 AVL 插入的完整实现

```cpp
AVLNode* insert(AVLNode* node, int val) {
    // 1. 普通 BST 插入
    if (node == nullptr) return new AVLNode(val);

    if (val < node->data)
        node->left = insert(node->left, val);
    else if (val > node->data)
        node->right = insert(node->right, val);
    else
        return node;  // 不允许重复

    // 2. 更新高度
    updateHeight(node);

    // 3. 计算平衡因子
    int balance = getBalance(node);

    // 4. 四种失衡情况

    // LL：左子树高，且插入在左子树的左边
    if (balance > 1 && val < node->left->data)
        return rightRotate(node);

    // RR：右子树高，且插入在右子树的右边
    if (balance < -1 && val > node->right->data)
        return leftRotate(node);

    // LR：左子树高，但插入在左子树的右边
    if (balance > 1 && val > node->left->data) {
        node->left = leftRotate(node->left);
        return rightRotate(node);
    }

    // RL：右子树高，但插入在右子树的左边
    if (balance < -1 && val < node->right->data) {
        node->right = rightRotate(node->right);
        return leftRotate(node);
    }

    return node;  // 无失衡，直接返回
}
```

### 3.11 AVL 删除

删除比插入更复杂，因为删除后可能需要**沿路径向上多次旋转**（插入最多只需一次旋转/双旋转）。

```cpp
AVLNode* deleteNode(AVLNode* node, int key) {
    // 1. 普通 BST 删除
    if (node == nullptr) return nullptr;

    if (key < node->data) {
        node->left = deleteNode(node->left, key);
    } else if (key > node->data) {
        node->right = deleteNode(node->right, key);
    } else {
        // 找到要删除的节点
        if (!node->left || !node->right) {
            AVLNode* temp = node->left ? node->left : node->right;
            if (temp == nullptr) {
                temp = node;
                node = nullptr;
            } else {
                *node = *temp;
            }
            delete temp;
        } else {
            // 度为 2：用中序后继替换
            AVLNode* temp = node->right;
            while (temp->left) temp = temp->left;
            node->data = temp->data;
            node->right = deleteNode(node->right, temp->data);
        }
    }

    if (node == nullptr) return nullptr;

    // 2. 更新高度
    updateHeight(node);

    // 3. 检查平衡并旋转
    int balance = getBalance(node);

    // LL
    if (balance > 1 && getBalance(node->left) >= 0)
        return rightRotate(node);

    // LR
    if (balance > 1 && getBalance(node->left) < 0) {
        node->left = leftRotate(node->left);
        return rightRotate(node);
    }

    // RR
    if (balance < -1 && getBalance(node->right) <= 0)
        return leftRotate(node);

    // RL
    if (balance < -1 && getBalance(node->right) > 0) {
        node->right = rightRotate(node->right);
        return leftRotate(node);
    }

    return node;
}
```

依次插入 {50, 30, 70, 20, 40, 10}：

```
插入 50：       插入 30：        插入 70：
   50              50               50
                  /               /    \
                 30              30     70

平衡 ✓          平衡 ✓          平衡 ✓

插入 20：         插入 40：
     50               50
    / \               / \
   30  70            30  70
  /                 / \
 20                20  40

平衡 ✓            平衡 ✓

插入 10：
       50    (BF=2, LL 型！)
      / \
     30  70  (BF=1)
    / \

  /
 10           ← 触发 LL 失衡

对 50 进行右旋：
       30
      /  \

所有节点 BF ∈ {-1, 0, 1}，平衡恢复 ✓
```

### 3.13 AVL 树性能分析

| 维度 | 复杂度 |
|------|--------|
| 查找 | O(log n)（保证） |
| 插入 | O(log n)（最多 1~2 次旋转） |
| 删除 | O(log n)（可能需要 O(log n) 次旋转） |
| 空间 | O(n) |

**AVL 树 vs 普通 BST**：

| 维度 | 普通 BST | AVL 树 |
|------|---------|--------|
| 最坏查找 | O(n) | O(log n) |
| 插入后调整 | 无 | 可能旋转（常数次） |
| 删除后调整 | 无 | 可能旋转（O(log n) 次） |
| 适用场景 | 数据随机分布 | 需要严格保证性能 |

## Part 4 ｜ 红黑树（Red-Black Tree）

## 第四章 红黑树

### 4.1 为什么有了 AVL 还需要红黑树？

AVL 树的**平衡条件太严格**——任意节点左右子树高度差不超过 1。这意味着：

- 插入/删除后经常需要旋转来恢复平衡
- 删除时可能需要 O(log n) 次旋转

红黑树放宽了平衡条件——不要求严格平衡，只要求“大致平衡”。结果：

| 维度 | AVL 树 | 红黑树 |
|------|--------|--------|
| 平衡程度 | 严格平衡 | 大致平衡（最长路径 ≤ 2 × 最短路径） |
| 查找效率 | 略优（树更矮） | 略差（但仍是 O(log n)） |
| 插入调整 | 可能旋转 | 最多 2 次旋转 + 变色 |
| 删除调整 | 可能 O(log n) 次旋转 | 最多 3 次旋转 + 变色 |
| 实际应用 | 查找密集场景 | 插入/删除频繁场景 |

> **工程中红黑树更常见**：C++ STL 的 `map`/`set`、Java 的 `TreeMap`/`TreeSet`、Linux 内核的进程调度等都使用红黑树。

### 4.2 红黑树的定义（五条性质）

红黑树是一棵 BST，每个节点被染成**红色**或**黑色**，且满足以下五条性质：

| 编号 | 性质 | 直觉理解 |
|------|------|----------|
| ① | 每个节点非红即黑 | 颜色只有两种 |
| ② | **根节点是黑色** | 树的“起点”是黑色 |
| ③ | **叶子节点（NIL）是黑色** | 这里的“叶子”是指空指针（哨兵节点），不是普通叶子 |
| ④ | **红色节点的孩子必须是黑色**（不能有两个连续的红节点） | "红不邻红" |
| ⑤ | **从任意节点到其所有后代叶子（NIL）的路径上，黑色节点数相同** | "黑路同" |

性质 ⑤ 中，从根到叶子路径上的黑色节点数称为**黑高（Black Height）**。

```
一棵合法的红黑树（B=黑色，R=红色）：

          13(B)
         /    \
       8(R)   17(R)
      / \     / \
    1(B) 11(B) 15(B) 25(B)
      \             /
      6(R)        22(R)

验证性质 ⑤（从根到每个 NIL 的黑色节点数）：
  13→8→1→NIL          黑节点数 = 3 (13, 1, NIL)
  13→8→1→6→NIL        黑节点数 = 3 (13, 1, NIL)
  13→8→11→NIL         黑节点数 = 3
  13→17→15→NIL        黑节点数 = 3
  13→17→25→22→NIL     黑节点数 = 3
  13→17→25→NIL        黑节点数 = 3
  所有路径黑节点数 = 3 ✓
```

### 4.3 红黑树的关键性质推论

由五条性质可以推导出：

> **红黑树从根到叶子的最长路径不超过最短路径的 2 倍。**

**证明**：
- 最短路径：全是黑色节点 → 长度 = 黑高 $h_b$
- 最长路径：黑红交替 → 长度 = $2h_b$（每个黑节点后面跟一个红节点）

因此红黑树的高度 $h \leq 2\log_2(n+1)$，查找、插入、删除都是 **O(log n)**。

### 4.4 红黑树的节点结构

```cpp
enum Color { RED, BLACK };

struct RBNode {
    int data;
    Color color;
    RBNode* left;
    RBNode* right;
    RBNode* parent;

    RBNode(int val)
        : data(val), color(RED), left(nullptr), right(nullptr), parent(nullptr) {}
};
```

> 新插入的节点默认为**红色**——因为插入红节点不会违反性质 ⑤（黑路同），只可能违反性质 ④（红不邻红），修复起来更简单。

### 4.5 插入操作的调整

插入新节点（红色）后，如果父节点也是红色，就违反了性质 ④。需要根据叔叔节点的颜色来调整。

设新插入节点为 N，父节点为 P，祖父为 G，叔叔（P 的兄弟）为 U：

#### 情况 1：叔叔 U 是红色 → 变色

```
调整前：                      调整后：
      G(B)                       G(R)  ← 变红（可能继续向上调整）
     / \                        / \
   P(R) U(R)    →            P(B) U(B)  ← 都变黑
   /                         /
  N(R)                      N(R)

操作：P、U 变黑，G 变红，然后把 G 当作新插入节点继续向上检查
```

#### 情况 2：叔叔 U 是黑色，且 N 是 P 的外侧孩子 → 单旋转 + 变色

```
情况 2a（LL 型，N 是 P 的左孩子，P 是 G 的左孩子）：

      G(B)                      P(B)
     / \         右旋 G        /   \
   P(R) U(B)    ————→       N(R)  G(R)
   /                                \
  N(R)                              U(B)

操作：P 变黑，G 变红，对 G 右旋

情况 2b（RR 型）是镜像的，左旋即可。
```

#### 情况 3：叔叔 U 是黑色，且 N 是 P 的内侧孩子 → 双旋转 + 变色

```
情况 3a（LR 型，N 是 P 的右孩子，P 是 G 的左孩子）：

      G(B)             G(B)               N(B)
     / \    左旋 P    / \     右旋 G     /   \
   P(R) U(B)  →    N(R) U(B)   →     P(R)  G(R)
    \              /                          \
    N(R)         P(R)                         U(B)

操作：先对 P 左旋（转化为情况 2），再对 G 右旋 + 变色

情况 3b（RL 型）是镜像的。
```

### 4.6 插入调整总结

| 情况 | 条件 | 操作 |
|------|------|------|
| 0 | 父节点是黑色 | 无需调整 |
| 1 | 叔叔是红色 | P、U 变黑，G 变红，向上继续检查 |
| 2 | 叔叔是黑色，N 在外侧 | 单旋转 + 变色 |
| 3 | 叔叔是黑色，N 在内侧 | 双旋转 + 变色（先转化为情况 2） |

**最后别忘了：根节点永远染黑（性质 ②）。**

### 4.7 红黑树 vs AVL 树 vs BST

| 维度 | BST | AVL 树 | 红黑树 |
|------|-----|--------|--------|
| 平衡保证 | 无 | 严格（高度差 ≤ 1） | 宽松（最长 ≤ 2 × 最短） |
| 查找最坏 | O(n) | O(log n) | O(log n) |
| 插入旋转 | 0 | ≤ 2 次 | ≤ 2 次 |
| 删除旋转 | 0 | O(log n) 次 | ≤ 3 次 |
| 树高上界 | n | 1.44 log n | 2 log n |
| 适用场景 | 数据随机 | 查找多、改动少 | 改动频繁（通用） |

> **考研角度**：红黑树主要考察定义和性质，完整的插入/删除代码实现要求较少。重点记住五条性质、理解为什么插入节点默认红色，并能判断一棵树是否满足红黑树性质。

## Part 5 ｜ B 树与 B+ 树

## 第五章 B 树（B-Tree）

### 5.1 为什么需要 B 树？

前面介绍的 BST、AVL、红黑树都是**二叉树**——每个节点最多 2 个孩子。它们适合在**内存**中使用。

但对于**磁盘存储**（数据库、文件系统），情况完全不同：

```
内存访问：~100 ns
磁盘访问：~10 ms（约慢 10 万倍）
```

**磁盘的特点**：每次读取数据以“块”（block，通常 4KB）为单位，读一个字节和读 4KB 的时间几乎相同。

因此，减少磁盘访问次数（即减少树的高度）比减少比较次数更重要。

**B 树的核心思想**：让每个节点存储**多个关键字**、拥有**多个孩子**，从而降低树的高度。

```
二叉树（高度 ≈ 20 层存百万数据）：
  磁盘读 20 次 → 20 × 10ms = 200ms

B 树（m=200，高度 ≈ 3 层存百万数据）：
  磁盘读 3 次 → 3 × 10ms = 30ms
```

### 5.2 B 树的定义

**$m$ 阶 B 树**（$m \geq 2$）是一棵满足以下条件的多路平衡查找树：

| 编号 | 性质 |
|------|------|
| ① | 每个节点最多有 $m$ 棵子树（即最多 $m-1$ 个关键字） |
| ② | 根节点至少有 2 棵子树（除非整棵树只有根节点） |
| ③ | 除根节点外，每个节点至少有 $\lceil m/2 \rceil$ 棵子树（即至少 $\lceil m/2 \rceil - 1$ 个关键字） |
| ④ | 所有**叶子节点**在同一层（保持多路平衡） |
| ⑤ | 每个节点中的关键字**从左到右递增排列** |

> 这里的“叶子节点”在王道体系中通常指最底层的**失败节点**（查找失败时到达的虚拟节点），而不是最底层的实际节点。

**以 5 阶 B 树为例**（每个节点最多 4 个关键字、5 棵子树）：

```
                    [30 | 60]
                   /    |    \
          [10|20]    [40|50]    [70|80|90]
```

每个节点内部的关键字有序：10 < 20，30 前面的子树都 < 30，30 和 60 之间的子树关键字在 30~60 之间。

### 5.3 B 树节点结构

```
一个 m 阶 B 树的节点结构：

┌────┬────┬────┬────┬─...─┬──────┬────┐
│ P0 │ K1 │ P1 │ K2 │     │ Kn   │ Pn │
└────┴────┴────┴────┴─...─┴──────┴────┘

- Ki (1≤i≤n)：关键字，K1 < K2 < ... < Kn
- Pi (0≤i≤n)：子树指针
  - P0 指向的子树中所有关键字 < K1
  - Pi 指向的子树中所有关键字在 Ki 和 K(i+1) 之间
  - Pn 指向的子树中所有关键字 > Kn
- n：当前节点的关键字个数
  - 根节点：1 ≤ n ≤ m-1
  - 其他节点：⌈m/2⌉-1 ≤ n ≤ m-1
```

### 5.4 B 树的查找

类似于在 BST 中查找，但每个节点内部需要做一次**多路比较**（可以顺序查找或二分查找）：

```
在下面的 5 阶 B 树中查找 42：

                    [30 | 60]
                   /    |    \
          [10|20]  [35|40|50]  [70|80|90]

第 1 步：在根节点 [30|60] 中查找
        30 < 42 < 60 → 走中间子树 P1

第 2 步：在 [35|40|50] 中查找
        40 < 42 < 50
        走 P2 指针 → 到达叶子层
        查找失败（42 不在树中）
```

### 5.5 B 树的插入

**核心规则**：新关键字总是插入到**最底层的非叶子节点**。如果插入后节点关键字数超过 $m-1$，需要**分裂**。

**分裂规则**：
1. 取该节点的**中间关键字**（第 $\lceil m/2 \rceil$ 个）
2. 中间关键字**上升**到父节点
3. 左边的关键字留在原节点，右边的关键字放入新节点

**以 5 阶 B 树为例**（每个节点最多 4 个关键字）：

```
依次插入：10, 20, 30, 40, 50, 60, 70, 80

插入 10, 20, 30, 40：
  [10 | 20 | 30 | 40]    已满（4 个关键字）

插入 50 → 溢出，需要分裂：
  [10 | 20 | 30 | 40 | 50]    5 个关键字，超过上限

  中间关键字 = 30（第 ⌈5/2⌉ = 第 3 个）
  30 上升成为新的根：

         [30]
        /    \
    [10|20]  [40|50]

插入 60：放入右边节点
         [30]
        /    \
    [10|20]  [40|50|60]

插入 70：放入右边节点，刚好满
         [30]
        /    \
    [10|20]  [40|50|60|70]  ← 满了

插入 80：右边节点溢出
         [30]
        /    \
    [10|20]  [40|50|60|70|80]  ← 溢出

  分裂 [40|50|60|70|80]，中间是 60，上升到父节点：

         [30 | 60]
        /    |    \
    [10|20] [40|50] [70|80]
```

### 5.6 B 树的删除

B 树的删除需要保证删除后每个节点的关键字数不低于 $\lceil m/2 \rceil - 1$。

删除分两大类：

#### 删除关键字在终端节点（最底层非叶节点）

直接删除即可。如果删除后关键字数不足，需要**借**或**合并**：

1. **向兄弟借**：如果左/右兄弟有多余的关键字，通过父节点“旋转”借一个过来
2. **合并**：如果兄弟都没有多余的，就和一个兄弟及父节点中的分隔关键字合并

```
5 阶 B 树，删除 70：

         [30 | 60]                   [30 | 60]
        /    |    \      →          /    |    \
    [10|20] [40|50|55] [70|80]  [10|20] [40|50|55] [80]

关键字数至少为 ⌈5/2⌉-1 = 2 → 还剩 1 个，低于下限。

向右兄弟无法借（假设没有右兄弟），向左兄弟 [40|50|55] 可以借：
  父节点的 60 下来，兄弟的 55 上去：

         [30 | 55]
        /    |    \
    [10|20] [40|50]  [60|80]
```

#### 删除关键字在非终端节点

用该关键字的**直接前驱**（左子树最大值）或**直接后继**（右子树最小值）替换，然后转化为删除终端节点中的关键字。

### 5.7 B 树的高度分析

含有 $n$ 个关键字的 $m$ 阶 B 树：

$$\log_m(n+1) \leq h \leq \log_{\lceil m/2 \rceil}\frac{n+1}{2} + 1$$

**实例**：$m = 200$，$n = 1,000,000$

- 最小高度 ≈ $\log_{200}(1000001) \approx 3$
- 最大高度 ≈ $\log_{100}(500001) + 1 \approx 4$

通常只需 3~4 次磁盘访问就能在百万数据中找到目标。

## 第六章 B+ 树

### 6.1 B+ 树 vs B 树

**B+ 树**是 B 树的变种，是**数据库索引中最常用的数据结构**（MySQL InnoDB 引擎就用 B+ 树）。

主要区别：

| 维度 | B 树 | B+ 树 |
|------|------|-------|
| 关键字存储 | 每个节点都存数据 | **只有叶子节点存数据**，内部节点只存索引 |
| 叶子节点 | 互不相连 | 用**链表**依次串联 |
| 关键字重复 | 不重复 | 内部节点的关键字会在叶子节点中重复出现 |
| 查找路径 | 可能在非叶子节点命中 | 必须查到叶子节点才能获得数据 |
| 子树数 | $n$ 个关键字 → $n+1$ 棵子树 | $n$ 个关键字 → $n$ 棵子树（定义不同） |

### 6.2 B+ 树的结构

```
                     [30 | 60]           ← 内部节点（只存索引）
                    /    |    \
             [10|20] [30|40|50] [60|70|80|90]  ← 叶子节点（存实际数据）
               │─────────│──────────│
               └──→  链表串联   ←──┘

内部节点 [30|60]：
  - < 30 的走左子树
  - ≥ 30 且 < 60 的走中间子树
  - ≥ 60 的走右子树

叶子节点之间有链表指针 →
  可以从 [10|20] 顺着链表走到 [30|40|50] 再到 [60|70|80|90]
```

### 6.3 B+ 树的优势

#### 优势 1：范围查询高效

```
查询“25 到 55 之间的所有数据”：

B 树：需要中序遍历，可能在树中上下反复跳
B+ 树：找到 25 所在的叶子节点，然后顺着链表往右扫描到 55，一路顺序读取！

           [30 | 60]
          /    |    \
    [10|20] [30|40|50] [60|70|80|90]
      ↑         ↑
    25落在这   55落在这
      ←────顺序扫描────→
```

#### 优势 2：内部节点更小，一个磁盘块可以放更多索引

B 树的内部节点既存关键字又存数据，而 B+ 树的内部节点只存关键字（索引）。同一个磁盘块能容纳更多关键字，树的阶更大，高度更低。

#### 优势 3：查询性能稳定

B 树的查找可能在第一层命中，也可能在最后一层命中，查询时间不稳定。
B+ 树的查找必须走到叶子节点，每次查询的 IO 次数相同，性能可预测。

### 6.4 B 树 vs B+ 树总结

| 维度 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储位置 | 所有节点 | 仅叶子节点 |
| 叶子链表 | 无 | 有 |
| 范围查询 | 慢（需遍历树） | 快（顺序扫描叶子链表） |
| 等值查询 | 可能更快（非叶子就命中） | 必须到叶子 |
| 内部节点大小 | 大（含数据） | 小（仅索引），同一磁盘块能放更多 |
| 典型应用 | MongoDB 索引 | MySQL/PostgreSQL 索引 |
